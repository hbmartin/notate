# Notate PP-OCRv3 — Comprehensive Code Review

**Branch:** `codex/ppocrv3-recognition-search`
**Date:** 2026-07-16
**Scope:** The new `:ocr-runtime` Paddle Lite library, stroke rasterization, background indexing, FTS search, and the WorkManager wiring — plus the core persistence paths the feature leans on.
**Method:** 14 review cells (12 fanned-out + 2 targeted deep-dives), each finder reading real source; every material finding independently verified by an adversarial skeptic pass; the top ~12 re-verified by hand against source. Build compiles; all 14 OCR unit tests pass.

---

## Verdict — Not release-ready; a robustness pass, not a redesign

The foundation is genuinely solid, and the seams that would have been catastrophic are correct: the CTC greedy decode, the flat `float[]` JNI (de)serialization contract, Kotlin-side dictionary bounds-checking, process-wide engine serialization via a coroutine `Mutex`, the raster↔world coordinate inverse, and the entire download / SHA-256 / atomic-activate security path all verified clean. There is **no critical memory-corruption or data-loss defect on the common, mutex-serialized ARGB_8888 path**, and the design doc is exceptionally accurate against the code.

What holds it back is a cluster of reliability, native-memory, and data-integrity defects — mostly in the background-indexing and worker layer — plus one HIGH search-correctness bug that silently returns zero results for realistic CJK queries. The single most dangerous item overall is a pre-existing, non-OCR data-loss window in the core atomic-commit path. None of this is architectural.

## Fix before shipping (blocker shortlist)

1. **Free the native contour arrays** — `boxes_from_bitmap` leaks `get_mini_boxes`' heap arrays on every contour → unbounded native RSS growth during background indexing. `H1`
2. **Fix the core atomic-commit window** — a crash between the two renames destroys the note and the `.bak` is never restored. Pre-existing, but the most severe integrity issue found. `H2`
3. **Fix CJK/mixed search verification** — realistic queries silently return nothing on a headline feature. `H3`
4. **Stop re-OCRing empty regions every pass** and make backfill drain poisoned work instead of looping forever. `H4 · M2`
5. **Un-latch `unavailableReason`** so one transient init failure doesn't disable all OCR for the session, and **stop purging a project's whole index** on a transient scan failure. `M1 · M3`

## High-severity findings

Wrong results, crashes, leaks, or data loss on plausible input. All independently verified against source.

### H1 — Native heap leak: DB post-processor never frees the per-contour `float**` arrays
**File:** `ocr-runtime/src/main/cpp/ocr_db_post_process.cpp:68` · `:153` · `:242–306`
**Category:** memory-safety / native

- **What:** `Mat2Vec()` does `new float*[]` + per-row `new float[]`; `get_mini_boxes()` returns it and is called *twice per contour* inside `boxes_from_bitmap()`. There is **no `delete`/`free` anywhere in the file** — every path, including all early `continue`s, leaks.
- **Impact:** Background indexing runs the detector over up to 1,000 contours per 960px tile, across many tiles and notes, in one long-lived process. RSS grows unbounded → native OOM risk on memory-constrained e-ink hardware.
- **Fix:** `delete[]` both arrays on every path, or refactor `Mat2Vec` to return a `std::vector` / RAII wrapper. Inherited from the upstream PaddleOCR demo.

### H2 — Non-atomic commit window destroys the note on a crash; the `.bak` is never restored
**File:** `app/src/main/java/com/alexdremov/notate/data/io/AtomicContainerStorage.kt:135 → :144` *(pre-existing core)*
**Category:** data-integrity / atomicity

- **What:** The "atomic" commit is `target.renameTo(backup)` then `source.renameTo(target)`. Between the two, the target path has *no* file; on success the `.bak` is deleted (`:163`) and **no open path ever reads `.bak` back**. `commitToSaf` is worse — it truncates (`"wt"`) with no backup at all.
- **Impact:** `SaveWorker` runs this during app teardown — exactly when process death is likely. A kill in the window leaves an orphan `.bak`; next open sees a missing file and creates a blank session → silent note loss. Narrow window, but real and unrecoverable.
- **Fix:** Restore `.bak` on open when the target is missing, or commit via a single `rename`-over-target (atomic on the same filesystem). Not part of the OCR branch, but the highest-impact integrity issue found.

### H3 — CJK/mixed search verify demands the whole query as one contiguous substring
**File:** `app/src/main/java/com/alexdremov/notate/ocr/index/OcrSearchRepository.kt:40–41`
**Category:** correctness / search

- **What:** FTS matches CJK order-independently via per-token unigram/bigram `AND` terms, but any CJK-containing query is then verified with `normalizedText.contains(entireQuery)` — which keeps internal spaces, punctuation, and word order.
- **Impact:** A CJK phrase typed with a space, or `"note <cjk>"` vs a doc containing `"<cjk> note"`, returns **zero results** on a headline feature — silently and hard to diagnose. The sole HIGH-severity finding from the adversarial pass.
- **Fix:** Verify per segment: each maximal CJK run and each non-CJK token as an independent substring, or strip separators from both sides before `contains()`.

### H4 — Regions that recognize to nothing are decoded & re-OCR'd on *every* index pass
**File:** `OcrIndexWriter.kt:82` · `:101` + `OcrIndexDao.kt:109` (empty insert skipped)
**Category:** incremental-correctness

- **What:** The "unchanged" check reads a region's hash from a stored block row. When a region yields *zero* blocks, `replaceRegion(…, emptyList())` deletes old rows and inserts nothing, so no `regionHash` is persisted. Next run `getRegionHash` returns `null ≠ hash` → decode + full re-OCR again, forever.
- **Impact:** Any region of scribbles, struck-out ink, diagrams, or blank text (anything below the 0.5 confidence gate) defeats the entire incremental design — the expensive tile rasterize + native inference re-runs on it on every background pass, and it's even reported as freshly `indexed`.
- **Fix:** Persist a per-region hash sentinel even when a region produces no blocks (e.g. a zero-block marker row keyed by `documentId,regionId`).

## Medium-severity findings

Real reliability, concurrency, and data-integrity issues on plausible-but-narrower paths. The worker layer carries the largest concentration.

### M1 — One transient init failure permanently disables all OCR for the process session
**File:** `PaddleLiteOcrEngine.kt:84–89` (set) · `:23` · `:25–28` (never cleared)

- **What:** `ensureLoaded()` caches *any* `Throwable` in `unavailableReason`; `isAvailable()` then returns false forever. It is only ever reset by creating a whole new engine, which happens solely on model install/remove.
- **Impact:** The engine is a process-wide singleton, so one transient hiccup (native OOM during cold init, or `prepare()` racing an atomic model swap) kills both manual *and* background OCR until the app is restarted.
- **Fix:** Only latch clearly-permanent failures (unsupported ABI, models-not-installed); leave `unavailableReason` null for transient errors so the next call retries.

### M2 — Poisoned region → permanent STALE + non-draining backfill retry loop
**File:** `OcrBackfillWorker.kt:48–67` + `OcrIndexWriter.kt:87–99` (STALE without storing hash)

- **What:** A single failing ink region marks the whole document `STALE` without storing its hash, so it re-OCRs every pass and never reaches `INDEXED`. The retry decision uses the pre-processing pending-count with no `runAttemptCount` cap, and `take(BATCH_SIZE)` always takes the head of a stably-ordered list.
- **Impact:** With more than 3 permanently-failing docs, the worker returns `Result.retry()` indefinitely (charging + idle, survives restart), burning battery re-OCRing the same docs while later ones may never index.
- **Fix:** Advance past already-failed docs (cursor/exclusion), gate retry on forward progress, and add a per-doc failure cap.

### M3 — A transient scan failure for one project purges that project's entire OCR index
**File:** `OcrBackfillWorker.kt:32–46`

- **What:** Each project scan is `runCatching{…}.getOrElse{ emptyList() }`, and `currentPaths` is a single global set. Every indexed document whose path is absent from that set is deleted — so a project that failed to scan contributes no paths and has all its docs/blocks removed.
- **Impact:** A momentary SAF/IO/provider hiccup wipes every `OcrDocument`/`OcrBlock` for that project though the files still exist. Search returns nothing for those notes until a slow (3/run) re-index — which per M2 may never converge.
- **Fix:** Track per-project scan success; only prune paths for projects that scanned successfully. Distinguish "empty" from "failed".

### M4 — `openCanvasSession()` throws outside the try → one locked note aborts the whole backfill run
**File:** `OcrBackfillWorker.kt:53` (call) vs `:58` (try) · `CanvasRepository.kt:137` (throws `CanvasLockedException`)

- **What:** On lock contention `openCanvasSession` throws `CanvasLockedException` after 10 retries — it does not return null. The call sits *outside* the per-item `try/catch`, so the exception propagates out of `doWork`; a `CoroutineWorker` that throws is terminal `Result.failure()`. The `session == null` guard is dead for the lock case.
- **Impact:** A note transiently locked by a concurrent `SaveWorker` aborts the entire backfill batch; indexing stalls until something reschedules it. (Found independently by two review passes.)
- **Fix:** Move `openCanvasSession` inside the per-item `try` (or `runCatching`) and `return@forEach` on failure.

### M5 — Unguarded `readBytes()` races non-atomic region writes → whole index run aborts
**File:** `OcrIndexWriter.kt:80` · `RegionStorage` writeAtomic (delete-then-rename)

- **What:** `indexSession` scans the *live* session directory and reads each region with `file.readBytes()` with no `try/catch` (only decode/build are guarded). Region writes are non-atomic delete-then-rename, leaving a window where the target is absent.
- **Impact:** A chained/backfill worker hitting that window throws `FileNotFoundException`, aborting the whole run and discarding all OCR compute so far; the doc set `STATUS_INDEXING` stays absent from search until a retry.
- **Fix:** Wrap per-file read/hash in `runCatching` and skip-as-stale; index a consistent snapshot; make region writes rename-over-target.

### M6 — Manual "Recognize text" squishes the whole selection into one ≤960px bitmap — no tiling
**File:** `app/src/main/java/com/alexdremov/notate/ui/OnyxCanvasView.kt` (recognizeSelection) · `StrokeOcrRasterizer`

- **What:** The manual path unions all selected strokes and scales to ≤960px, unlike the background path which deliberately tiles at 960px / stride 864 because PP-OCRv3 needs adequate text height.
- **Impact:** A full-page lasso scales ~50px handwriting down to ~18px, so the primary user-facing action returns "No text recognized" even though background indexing of the same strokes succeeds. (Recognition-quality at reduced height is native-lib behavior; the path divergence is certain.)
- **Fix:** Reuse the tiling helper (`recognizeTiles`) for the manual path so large selections recognize tile-by-tile and merge.

### M7 — Block + FTS rows orphaned when a document's ID changes at a stable path
**File:** `OcrIndexWriter.kt:49` · `:55` · `OcrIndexDao.kt:29` (REPLACE) · `OcrIndexDatabase.kt:30–37` (no FK)

- **What:** `ocr_blocks` has no foreign key/cascade, `upsertDocument` is `@Insert(REPLACE)`, and `ocr_documents` has `UNIQUE(path)`. A note first indexed as `sha256(path)` (no UUID) that later gains a UUID re-keys the document; REPLACE deletes the old row on the path conflict and its blocks are orphaned. `search` JOINs documents (`:155`) so orphans are invisible, and backfill cleanup only prunes by missing path — never orphan blocks.
- **Impact:** A growing, unreachable block+FTS leak on every ID-changing event. (The missing cascade is load-bearing elsewhere — it's why the `INDEXING` REPLACE at `:55` doesn't wipe the note's own blocks — so the fix is explicit orphan cleanup, not a naive FK.)
- **Fix:** Add an explicit orphan-block sweep (blocks whose `documentId` has no document row), or reconcile identity before REPLACE.

### M8 — Backfill and the save-chain can index the same document with no mutual exclusion
**File:** `OcrIndexWriter.kt:105–107` (snapshot cleanup, no lock) · `OcrBackfillWorker` vs `CanvasRepository` save chain

- **What:** The save chain (`SaveWorker_$name`) and `OcrBackfillWorker` are separate unique works that both call `indexSession` for the same path. Nothing serializes per-document; Room only serializes individual transactions. Removed-region cleanup deletes every DB region not in *one run's* snapshot.
- **Impact:** A backfill run working from a slightly older on-disk snapshot can delete a region another worker just correctly indexed — dropping indexed content — and the two runs are last-writer-wins on status. Timing-dependent (needs charging+idle overlapping a save).
- **Fix:** Serialize `indexSession` per `documentId` across both workers (a keyed mutex / unique-work coalescing).

### M9 — Onyx SDK native AARs pulled over cleartext HTTP with no dependency verification
**File:** `settings.gradle.kts:18` (allowInsecureProtocol) — repo.boox.com *(pre-existing core)*

- **What:** `onyxsdk-pen`/`onyxsdk-base` ship in-process native `.so` and are fetched from `http://repo.boox.com` with `isAllowInsecureProtocol = true`. There is no `verification-metadata.xml` / lockfile, so nothing checks binary integrity.
- **Impact:** A MITM or repo/DNS compromise can serve a tampered `.so` that Gradle packages into the signed release APK with the app's full permissions. Pre-existing on `main`, but in scope because it packages native code alongside the new OCR libraries.
- **Fix:** Use HTTPS if Onyx offers it; otherwise pin SHA-256 for every `com.onyx.android.sdk` artifact via Gradle dependency verification.

### M10 — Room index DB has no migration path and no destructive fallback
**File:** `OcrIndexDatabase.kt:63–81` (version = 1, exportSchema = false, plain `build()`)

- **What:** No `addMigrations(…)` and no `fallbackToDestructiveMigration()`; with `exportSchema = false` there's no schema JSON to author migrations against.
- **Impact:** The moment anyone bumps `version` to 2 without a `Migration`, Room throws on first open after update — an app-wide crash for every existing user. Latent (version is 1 today), and the doc itself flags this, but it's a footgun baked into the config.
- **Fix:** For a fully-rebuildable index/cache DB, `fallbackToDestructiveMigration()` is the safe default.

### M11 — Per-region transactions, not per-document — a mid-run kill leaves half-updated docs stuck `INDEXING`
**File:** `OcrIndexWriter.kt:55–66` (INDEXING up front) · `:101` (per-region commit) · `:108–120` (final status)

- **What:** Only individual regions are transactional. Status is set `INDEXING` immediately, regions commit one by one, and the final `INDEXED`/`STALE` is written last. `search` has no status filter.
- **Impact:** A kill (OOM / process death / WorkManager timeout) after some regions commit leaves a mix of new and old blocks and `status = INDEXING` forever; search serves that inconsistent mixture. Self-heals only if backfill (charging+idle) re-picks non-`INDEXED` docs.
- **Fix:** Add `isStopped` checkpoints and reset/repair `INDEXING` stragglers on the next run; consider filtering `INDEXING` out of search.

## Systemic themes

- **Inherited demo code carries latent native defects.** The heap leak, unchecked `lockPixels`, dead `origin.size` guard, ODR-duplicated converter, RGB-on-BGR normalization, and `LITE_POWER_LOW` map all trace to the upstream PaddleOCR/Paddle-Lite demo lineage. Route all bitmap conversion through one validated RAII implementation. *(H1, L1–L4, L12)*
- **Workers treat batches as all-or-nothing and never drain poison.** No forward-progress check, no per-item failure cap, no item-level fault isolation. One bad region/lock/scan aborts or re-loops the whole run. *(M2, M3, M4, M5, L5)*
- **Transient failures get latched or amplified into durable damage.** A recoverable hiccup becomes a permanent disable (M1), a whole-project index purge (M3), or a note stuck `INDEXING` (M11). The system lacks a "this might just be transient" distinction.
- **Non-atomic delete-then-rename file ops create races.** Both region writes and pack/commit activation delete-then-rename rather than rename-over-target, opening windows where the target is briefly absent — surfacing as H2, M5, and transient "not installed" states.
- **Manual and background OCR paths diverged.** Identical strokes recognize differently depending on entry point because the manual path skips the tiling the background path relies on. *(M6)*
- **CJK/Unicode is bolted onto ASCII-oriented tokenization.** The English path is correct; the CJK path mishandles spaces/word-order (H3) and astral code points (L6).

## What's verified correct

The high-stakes seams were adversarially checked and hold — the reason the verdict is "robustness pass," not "redesign."

- CTC greedy decode — blank skip, repeat-collapse, per-timestep argmax
- Flat `float[]` JNI layout — serialize/parse offsets match exactly
- Dictionary bounds-checked — `index in labels.indices`, no OOB
- Engine serialization — one Mutex over every native call; no use-after-free / double-free
- Raster ↔ world inverse — exact, and unit-tested
- Download integrity — SHA-256 on `.part` after `fsync`, correct file
- HTTPS enforced — cross-protocol redirects rejected
- Atomic model activation — staging → rename; resume distinguishes 206/200/416
- No executable download — native `.so` ship in the AAR only
- FTS injection-safe — MATCH is a bound parameter, tokens hex-escaped
- Region hash mixes model version — upgrades invalidate correctly
- ABI guard consistent — arm64-v8a filter + runtime check; no models in APK

## What static review can't confirm (needs the physical Snapdragon 750G)

The build compiles and all 14 OCR unit tests pass, but JVM/build checks don't exercise the model.

- **Recognition accuracy & latency** — whether the untiled manual path (M6), RGB-on-BGR normalization, 32-multiple under-sizing, and ~0.5 threshold actually hurt recall, on real handwriting.
- **Native RSS growth & OOM threshold** from the heap leak (H1), and whether `lockPixels` / cold Paddle Lite init actually fail transiently on target hardware (which is what makes L1 and the M1 latch reachable).
- **Race hit-rates** — the `readBytes`-vs-write window (M5) and install/swap TOCTOU are microsecond windows; the race exists, but not how often it fires under real autosave/eviction cadence.
- **WorkManager 10-minute window** — whether realistic multi-region notes exceed it (M11) is purely runtime-data-dependent.
- **On-device packaging** — 16 KB-page install, `pickFirst` surviving future Onyx SDK bumps, and downstream Maven OpenCV collisions weren't exercised against a shipped, installed artifact.
- **Test coverage is thin** — 14 OCR unit tests cover normalization, geometry, install, and one writer path, but none of the empty-region rehash (H4), orphaning (M7), worker retry/purge (M2/M3), or CJK-space-query (H3) defects. Add regression tests with the fixes.
